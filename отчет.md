# Отчет По Прогону 5 Презентативных Запросов

Дата: 2026-04-18  
Стенд: `n8n + Qdrant + Ollama` (локально)  
Источник метрик: `n8n execution` (взяты прогоны без cache-hit для честной демонстрации)

## 1) Exact Lookup (Сильная Сторона)

Запрос: `Какой статус у компании ТОО "НурТех Групп"?`  
Execution: `266`  
Маршрут: `queryMode=status_lookup`, `retrievalMode=metadata`, `cacheHit=false`  
Время выполнения: `27.0 ms`

Ответ:
`Активно`  
`Подтверждение: ТОО "НурТех Групп" | БИН: 687156468494 | Алматы | Активно`

Вывод: быстрый и точный ответ в metadata-only режиме.

## 2) Fuzzy/Опечатка По Названию (Сильная + Слабая Сторона)

Запрос: `Какой статус у нуртех груп?`  
Execution: `258`  
Маршрут: `queryMode=status_lookup`, `retrievalMode=metadata`, `cacheHit=false`  
Время выполнения: `33.0 ms`

Ответ:
`Активно`  
`Подтверждение: АО "НурТех Групп" ...; ТОО "НурТех Групп" ...`

Вывод: fuzzy-матч работает даже с опечаткой, но при одинаковом бренде в разных юрформах возвращается несколько сущностей (нужна явная дизамбигуация).

## 3) Count С Фильтрами (Сильная Сторона)

Запрос: `Сколько ликвидированных компаний в Павлодаре?`  
Execution: `259`  
Маршрут: `queryMode=count`, `retrievalMode=metadata`, `cacheHit=false`  
Время выполнения: `22.0 ms`

Ответ:
`4`  
`Подтверждение: примеры компаний со статусом "Ликвидировано" в Павлодаре`

Вывод: count-сценарий работает быстро и предсказуемо через metadata count.

## 4) Unsupported Поле (Сильная Сторона Guardrail)

Запрос: `Какая выручка у ТОО "НурТех Групп" за 2025 год?`  
Execution: `265`  
Маршрут: `queryMode=unsupported`, `retrievalMode=none`, `cacheHit=false`  
Время выполнения: `13.0 ms`

Ответ:
`Недостаточно данных в контексте. Доступны только поля: БИН, название, город, статус, вид деятельности, дата регистрации.`

Вывод: корректный ранний short-circuit, без ложных «галлюцинаторных» ответов.

## 5) Размытый Semantic Запрос (Слабая Сторона)

Запрос: `Подбери что-то про обслуживание оборудования, похожие компании`  
Execution: `272`  
Маршрут: `queryMode=list`, `retrievalMode=semantic`, `cacheHit=false`  
Время выполнения: `1028.0 ms`

Ответ:
Список компаний, включая:
`АО "Лидер Трейд", АО "DataLine Групп", АО "Достык Трейд", ...`

Вывод: semantic fallback работает, но заметно медленнее и местами менее релевантен по бизнес-ожиданию (пример: попадание компаний со спорной релевантностью для user intent).

## Сводка Для Демо

Сильные стороны:
- Exact/filter/count/unsupported сценарии работают быстро (`~13–33 ms` execution-time).
- Улучшен fuzzy lookup по названию (включая запросы без кавычек/юрформы и с опечаткой).
- Guardrail по unsupported полям предсказуем и безопасен.

Слабые стороны:
- Semantic путь пока сильно тяжелее (`~1.0s`), не укладывается в целевой low-latency сценарий.
- При совпадающих названиях бренда в разных юрформах не хватает встроенной логики уточнения (город/юрформа/БИН).
- Для semantic-ответов нужна дополнительная калибровка релевантности и ранжирования.

Рекомендация для следующего шага:
- Добавить короткий clarify-step при множественных кандидатах по названию и ужесточить пост-фильтрацию semantic результатов.
